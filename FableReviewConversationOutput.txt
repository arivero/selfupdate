  Quality scale:

  ## Top Priority Findings

  ### 1. Branch-law violation: tail-specific active surface is still everywhere

  Files affected

  - src/selfupdate/train/layerwise.py
  - src/selfupdate/config.py
  - configs/base.yaml
  - many configs/experiments/lw_*tail*, lw_*final*, lw_r_tailpure*
  - tests/test_tail_ce.py
  - tests/test_conn_window.py
  - tests/test_mixed_schedule.py
  - scripts/tail_graft.py
  - scripts/report.py
  - paper/make_figs.py

  Problem

  The branch says: no new active tail-only/top-k classical distillation. But the implementation still exposes the old public abstraction as tail_step, tail_ce_blocks, tail_ce_weight,
  tail_hidden_weight, tail_ce_kind. Even if some of that machinery now implements connected windows, the names and config API keep reintroducing the forbidden concept.

  Action

  - Rename implementation primitive: tail_step → window_step.
  - Rename config knobs:
      - tail_ce_blocks → top_readout_window or readout_window_blocks
      - tail_ce_weight → readout_weight
      - tail_hidden_weight → window_hidden_weight
      - tail_ce_kind → readout_source

  - Add validator: reject tail_ce_blocks > 0 unless conn_window > 0, conn_stride == 1, or run class is explicitly legacy_archive / baseline.
  - Move old tail configs/tests/scripts to archive or ../selfupdate_kd.

  ———

  ### 2. configs/base.yaml defeats the “UNSET sentinel” rule

  File

  - configs/base.yaml

  Current

  train:
    method: layerwise
    schedule: summed
    tail_ce_kind: teacher_kl

  Problem

  TrainConfig.tail_ce_kind defaults to 'UNSET', but base.yaml sets it to teacher_kl. Therefore experiments that do not pin it still inherit a default. That recreates the silent-default
  confound the repo explicitly warns about.

  Action

  - Remove tail_ce_kind / readout_source from configs/base.yaml.
  - Require every experiment with readout weight/window to pin it.
  - Add scripts/audit_configs.py and a test that fails inheritance.

  ———

  ### 3. teacher_censored says “NO readout CE” but still wires last_block_ce_weight

  File

  - src/selfupdate/train/layerwise.py

  Code path

  _censored_item():

  if L == n:
      loss_val, _ = last_block_step(
          ...
          cfg.train.last_block_ce_weight,
      )

  Problem

  The docstring says the restored teacher-censored schedule has “NO connected window and NO readout CE”, but the code still allows last_block_ce_weight. That is task-label supervision if
  set.

  Action

  - In _validate_knob_schedule, reject last_block_ce_weight > 0 and lens_ce_weight > 0 for teacher_censored, unless classified as baseline.
  - Or force ce_w=0.0 inside _censored_item.

  ———

  ### 4. Hot loop still syncs on every block

  File

  - src/selfupdate/train/layerwise.py

  Current syncs

  return loss.item(), h_out.detach()
  return [l.item() for l in losses], h.detach()

  Problem

  This is exactly the documented sync-bound bottleneck: .item() inside the per-block/per-item walk forces GPU-host synchronization many times per item.

  Action

  - Return detached GPU scalars or tensors.
  - Accumulate logging tensors on device.
  - Convert to Python floats only at grad-accum boundary or epoch boundary.

  ———

  ### 5. Scheduler ignores custom SCHED when writing scheduler pid

  File

  - scripts/gpu_scheduler.sh

  Current

  echo $$ > runs/.sched/scheduler.pid

  Problem

  The script creates "$SCHED" but writes the pid to hardcoded runs/.sched. In multi-node mode SCHED=runs/.sched-$(hostname -s), so this either writes to the wrong place or fails if
  runs/.sched does not exist.

  Action

  echo $$ > "$SCHED/scheduler.pid"

  ———

  ## Main trainer

  ### src/selfupdate/train/layerwise.py

  What it does

  Implements layerwise training schedules:

  - summed
  - sequential
  - teacher_censored
  - mixed
  - connected windows
  - online teacher
  - anchors
  - offloaded Adam
  - LoRA/frozen teacher handling
  - checkpoint save and vocab tripwire

  Quality: B overall

  Core mechanics are strong, but this file is now overloaded and branch-contaminated.

  Strong parts

  - Good frozen-vocabulary tripwire.
  - Clean OnlineTeacherSource.
  - Good locality discipline through detached inputs.
  - Useful optimizer offload utility.
  - Sliding window semantics are implemented and tested.

  Major issues

  1. Public API still tail-specific.
  2. _validate_knob_schedule() is too weak.
  3. teacher_censored still permits last_block_ce_weight.
  4. .item() hot-loop syncs remain.
  5. tail_step() mixes generic window semantics, top readout, hidden loss, teacher KL, and task-label CE in one function.
  6. anchor_step() is coupled to tail_ce_blocks.
  7. File is almost 1000 lines and should be split.

  Actionable refactor

  Split into:

  - train/window.py: generic window_step
  - train/schedules/summed.py
  - train/schedules/censored.py
  - train/schedules/sequential.py
  - train/readout.py: teacher KL / baseline task-label CE
  - train/anchors.py
  - train/online_teacher.py
  - train/validation.py

  Specific validation additions

  Reject these unless explicitly baseline/archive:

  - tail_ce_blocks > 0 and conn_window <= 0
  - tail_ce_blocks > 0 and conn_stride != 1
  - tail_ce_kind == "task_label" in method run
  - last_block_ce_weight > 0 in method run
  - lens_ce_weight > 0 and lens_ce_from > 1 in method run
  - depth-increasing readout weights
  - teacher_censored with any CE/readout knob
